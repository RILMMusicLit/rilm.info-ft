Structure

Each file will contain one article or encyclopedia entry.

The root element for each document is <TEI>.

Each file has a <teiHeader> element that includes general metadata about the contents of the file.

teiHeader: Most of the teiHeader can remain constant from article to article. The things that will change are:

  • the ID of the article in the <TEI> element. (The ID should now appear in the <text> element.)

  • The <date> and <biblScope> elements in the <sourceDesc>

  • The attributes and content of the <application> element in the <appInfo> element.

  • The <title> element. Specific publications will have inststructions for this.

  • Specific publications might have additional elements.

The <text> element that follows includes a <front> and a <body> element. The <text> element includes one @type attribute that identifies the type of article (in the MGG Personenteil: "person", "family", "corporate"; in the MGG Sachteil: "topic").

The <text> element also contains the article ID as an @xml:id attribute. That ID will consist of a three-letter code for the publication and an auto-generated number.

The <text> element must also contain an @xml:lang attribute.

All content within the <front> and<body> elements should be in <div> elements, and all <div> elements should have @type attributes. The value of @type will vary some from publication to publication, but possible values include: "heading", "bio", "works", "abstract", "ography". Each <div> element should receive an @xml:id attribute that starts with the letter d followed by a number. The first number will be 1, so the first <div> created in any article will have xml:id="d1".

The following elements should all have an xml:id attribute, numbered sequentially for each article: <div>, <p>, <bibl>, <rilmPerson>

Citations

Parenthetical citations should be set (inlcuding the parentheses) in a <ref> element that contains an @target attribute pointing to the appropriate bibliographic citation. Each item in the reference list should receive an @xml:id attribute to be used for this purpose.

For footnote or endnote citations, each reference number should be set in <ref> element

Names

All instances of <persName>, <author>, and <editor> should contain <forename> and <surname> when possible. (Some names will also require <nameLink>, <addName>, <genName>, or <roleName>) Abbreviated names should appear within <abbr>. For example, the name "D. Franković" might appear like this: <author><forename>< abbr>D.</abbr><forename> <surname>Franković</surname></ author>. If there are more than one any any type of name, repeat the elements as necessary: <forename>John</forename><forename>Paul</forename><surname>Jones</surname>.

The exception to this is surname prefixes, a list of which follows. Prefixes that start with a lower-case letter should be tagged <nameLink>. Prefixes that start with an upper-case letter should be considered part of the surname. Surname prefixes can have multiple words in them.

von van de van der von der van den van de ten ter op de
d’ di de la del de los de las da das do dos

Include only the following terms and abbreviations in <genName> (with any capitalization):

d.Ä. d.J. fils père jnr snr jr. sr.
jun. sen. I II III IV V [...]

Each <docAuthor> element, like each <rilmPerson> and each <author>, should contain only the names of only one person.

Dates

For all dates that should tagged with <date> and an @when attribute, the @when attribute can only contain a numerical date in w3c format, that is, dddd-dd-dd, dddd-dd, or dddd where d is a digit. Each of these dates should also receive an @when-custom attribute with only a four-digit year. Years earlier than 1000 CE should have a leading zero (e.g. when-custom="0863"). For approximate dates, add @precision="low". For more, see individual publications below.

Quotes and quotation marks

Quotation marks should be replaced by one of four elements appropriate to the content: <q>, <quote>, <soCalled>, or <title>. Generally, <quote> should be used for quoted text that appears with a citation, either within a <cit> element or with an @source attribute; <q> can be used as a replacement for quotation marks when the reason for those quotation marks is uncertain. However, see notes for specific publications.

For right- or left quotation marks that appear (usually because of a typo) without the corresponding mark, use <q> @rend and one of the following values: "right-double", "left-double", "right-single", "left-single".

The American Rule

For those publications which are marked "Use American rule": only mark  the single right-quote ( ’ ​)​ ​ character as <q> when it appears along with a single left-quote character and both are embedded in double quotes, as in the following:

 
Smith wrote, “Every singer in Germany knows ‘Die Lorelei,’ and can sing it in any key.”

Which could be converted as

Smith wrote, <q>Every singer in Germany knows <q>Die Lorelei,</q> and can sing it in any key.</q>

Block quotes

Block quotes should be marked within with following markup <cit rend="block"><quote>Lorem ipsum.</quote><cit>. When there is an attribution (a reference) at the end of the quote, it should be tagged <ref> after the end of <quote> but before the end of <cit>. If possible that reference should link to the appropriate footnote or entry in a works cited list.

For block quotes that include verse lines, such as the example below, mark the each group of lines in <lg> and each line in <l>.

Big Jim Lee up the river, backin up an down,
And the side wheel knocking, Lord, I’m water boun
you’d need to learn on your own that he was singing about the Mississippi riverboat James Lee when it got stuck on a sandbar.

In XML:

<cit rend="block"><quote>
<lg>
<l>Big Jim Lee up the river, backin up an down,</l>
<l>And the side wheel knocking, Lord, I’m water boun</l>
<l>you’d need to learn on your own that he was singing about the Mississippi riverboat James Lee when it got stuck on a sandbar.</l>
</lg>
</quote></cit>

Indexes (back-of-the-book)

Use the following method for indexes:

  1. If an index term points to single page numbers, find the first instance of that term on that page and tag it with <index>. NB: the term may be inverted, so for "Jackson, Michael" you would find the first instance of "Michael Jackson".
  2. If the index term points to a page range, find the first instance of that term on the first page of the range and tag it with <index>.
  3. See instructions for specific publications for more information.

Front- and Back Matter

Unless otherwise instructed in the instructions for the specific publication, create one XML file for each section of the front matter and back matter. Mark up those files up with formatting tags only: <hi>, <q>, <p>, <head>, <list>, <item>, <table> etc. and the text should be <text type="matter">. A section called "Preface" would receive one XML file, as would a section called "Acknowledgements" or a section called "Contents".

Divisions within articles

Within the lowest-level <div> elements, prose paragraphs are encoded in <p> elements and lists (whether in paragraph format or list format where each entry is on a new line) are encoded in <list>elements.

For sections within articles that are designated by a number, letter, or other symbol, the symbol should be moved from the text itself to an @n attribute. That attribute should appear in the appropriate <div> element. If there is a heading for that section, the name @n attribute should also appear in the <head> element.

Example:

<div type="section" xml:id="d5" n="III">

<head n="III">Geschichte der Querflöte</head>

<div type="section" xml:id="d6" n="1">

<head n="1">Frühgeschichte bis 1650</head>


Lists

Lists in the source documents are formatted sometimes with one item per line, and sometimes inline, with semicolons or other punctuation separating each item.

Within <list> elements, use <head> and <item> elements. If text is neither a heading or an item in the list, use <note>element.

When possible, put <head> elements outside of <list>, that is, use them in conjunction with <div> instead.

Example:

<div type="section" n="2" xml:id="d24"><head n="2">Klarinettenkonzerte</head> <list><item>...</item><item>...</item></list></div>

When an <item> element contains a work (in the case of a works list or a bibliography) and the bibliographic data relating to that work is able to be tagged, the immediate child of <item> should be <bibl>. <bibl> may then contain <title>, <publisher>, <date>, <pubPlace>, <opus>, <musKey>, <biblScope>, etc.

Example:

<head> Pour l'orgue, </head>
<list>
<item><bibl><title>3 Préludes et fugues</title>, <opus>op. 7</opus>, <pubPlace>Paris</pubPlace>, <publisher>Leduc</publisher>, <date>1912</date> ; </bibl></item>

Formatting and markup

Generally, within prose paragraphs, markup should be added only to reflect formatting, hyphenation, page breaks, and the like. In lists, where formatting indicates the parts of list items in a consistent manner, those items can be marked up semantically with elements such as <title>, <author>, and <date>. Tags such as <opus> and <date> may be able to be marked up based on the internal format of the data; that is, <opus> will aways begin with one of a predifined set of abbreviations followed by a string that begins with an Arabic numeral.

Fractions: Fractions should be encoded as arabic numerals separated by a fraction slash ( ⁄).

Time signatures: If recognizable through formatting, time signatures should be marked up as arabic numerals separated by a slash (or 𝄴 or 𝄵) within a <timeSig> element. E.g. <timeSig>3/4</timeSig>

For musical symbols, use ♯, ♭, and ♮ and Unicode range 1D100–1D1FF.

Page breaks and hyphenation

Page breaks and column breaks should be marked with <pb> and <cb> elements respectively and @n

When hyphens appear at the end of lines, they should be marked as <pcforce="inter">-</pc>.

When a hyphen appears at the end of a page or a column, add @break="no" to <pb> or <cb> element.

Figures

Each figure should be set in <figure>, <notatedMusic>, or <tableFigure>, which should surround one of the following: <graphic>,<formula>, or <table>. A caption should be set in <p>. If there is a label or heading on the caption ("Figure 3"), it should be set in <head>.

Cross-References

Cross-references—links from one part of a publication to another—should be encoded in a <ref> element. If the reference points to a whole article, the @target should point to the xml:id of that article In this case there should be no "#" in the @target contents, and the reference should be to the xml:id rather than the filename (which has a file extension):

müßte reichen von der <ref target="mgg15403">Frühchristlichen Musik</ref> bis zur geistlichen Musik
If the reference points to a section of another article, the "#" should be used to indicate that section:
<ref target="mgg15546#d9">Kammermusik im 20. Jahrhundert</ref>

Footnotes and endnotes

Occasionally, there will be footnotes or endnotes in a publication. The contents of each footnote should be placed at the position of the reference mark in the following format. The reference mark itself (which will often be a superscript numeral) can be deleted.

<note n="1" place="end">Lexikon des Blasmusikwesens, 1. Auflage, Freiburg im Breisgau 1973; weitere Auflagen ebda. 1976, 1988 und 1994, Blasmusikverlag Schulz.</note>
<note n="*" place="foot">Ibid, p.7</note>

Line breaks in XML

Allow line breaks in the XML code only beween block-level elements.
